# MythosEngine FastAPI Server Guide

## Overview

This is a FastAPI backend server that wraps the existing MythosEngine Python backend and exposes it as REST endpoints for the Electron/React frontend.

The server initializes an `AppContext` on startup, which provides access to all services (config, storage, auth, managers, AI, etc.) in a single service locator pattern.

## Project Structure

```
server/
├── __init__.py              # Package marker
├── app.py                   # Main FastAPI app, startup/shutdown, CORS, route registration
├── deps.py                  # Dependency injection (get_ctx, get_current_user, token store)
├── routes/
│   ├── __init__.py
│   ├── auth.py             # POST /auth/login, logout, register, change-password, GET /auth/me
│   ├── notes.py            # Notes & folders CRUD, search, move operations
│   ├── ai.py               # POST /ai/ask, summarize, suggest-tags
│   ├── dashboard.py        # GET /dashboard/stats, recent notes
│   ├── users.py            # Admin user management (list, update roles, enable/disable)
│   ├── settings.py         # GET/PUT /settings for app configuration
│   └── invites.py          # Admin invite code generation & management
└── GUIDE.md                # This file
```

## API Endpoints

### Authentication (`/auth`)

- `POST /auth/login` — Authenticate with email/password, get bearer token
- `POST /auth/logout` — Revoke token
- `GET /auth/me` — Get current user info
- `POST /auth/change-password` — Change password
- `POST /auth/register` — Register new account with invite code

### Notes & Folders (`/notes`)

- `GET /notes?vault=<id>&folder=<id>` — List notes, optionally filtered by folder
- `GET /notes/<id>` — Get note detail with content
- `POST /notes` — Create new note
- `PUT /notes/<id>` — Update note
- `DELETE /notes/<id>` — Delete note
- `GET /notes/search?q=<query>` — Search notes by title/tags
- `POST /notes/move` — Move note to different folder
- `GET /folders?vault=<id>` — List folders in vault
- `POST /folders` — Create folder
- `DELETE /folders/<id>` — Delete folder

### AI (`/ai`)

All AI endpoints require authentication. Returns `503 Service Unavailable` if AI engine not initialized.

- `POST /ai/ask` — Ask a question (with optional vault context)
- `POST /ai/summarize` — Summarize text
- `POST /ai/suggest-tags` — Suggest tags for text

### Dashboard (`/dashboard`)

- `GET /dashboard/stats` — Get counts: notes, folders, characters, sessions, vaults
- `GET /dashboard/recent?limit=10` — Get 10 most recently modified notes

### Users (`/users`) — Admin Only

- `GET /users` — List all users
- `PUT /users/<id>/roles` — Update user roles
- `POST /users/<id>/disable` — Disable user
- `POST /users/<id>/enable` — Enable user
- `POST /users/<id>/reset-password` — Admin password reset

### Settings (`/settings`)

- `GET /settings` — Get current settings (theme, font_size, model names, etc.)
- `PUT /settings` — Update settings (persisted to config file)

### Invites (`/invites`) — Admin Only

- `GET /invites` — List all invite codes
- `POST /invites` — Generate new invite code
- `DELETE /invites/<id>` — Revoke invite code

## Authentication

All endpoints except `/health`, `/auth/login`, and `/auth/register` require authentication.

Use Bearer token authentication:

```
Authorization: Bearer <token>
```

Tokens are generated by the login endpoint and stored in memory (via the `TokenStore` in `deps.py`).

## Dependency Injection

The server uses FastAPI's `Depends()` mechanism for DI:

- `get_ctx()` — Retrieves the global `AppContext` from app state
- `get_current_user()` — Retrieves the authenticated `User` from the Bearer token
- `get_token_store()` — Retrieves the in-memory token store
- `require_admin()` — Ensures user has admin role

Example:

```python
@router.get("/example")
async def example(
    ctx: AppContext = Depends(get_ctx),
    user: User = Depends(get_current_user),
):
    # user is automatically validated, ctx is injected
    return {"user": user.username}
```

## Token Management

Tokens are stored in memory via `TokenStore` in `deps.py`. For a local desktop app, this is sufficient.

Key methods:
- `store(token, user_id)` — Map a token to a user
- `get_user_id(token)` — Look up user by token
- `revoke(token)` — Revoke a single token
- `revoke_all_for_user(user_id)` — Logout user everywhere

## Error Handling

All routes use standard HTTP status codes:

- `200` — Success
- `201` — Created
- `400` — Bad request (validation error)
- `401` — Unauthorized (missing/invalid token)
- `403` — Forbidden (permission denied)
- `404` — Not found
- `500` — Server error
- `503` — Service unavailable (AI not initialized)

Error responses are JSON:

```json
{
  "detail": "Error message"
}
```

## Request/Response Models

All endpoints use Pydantic models for request/response validation. Models are defined in each route module.

Example request:

```json
POST /auth/login
{
  "email": "user@example.com",
  "password": "password123"
}
```

Example response:

```json
{
  "token": "...",
  "user": {
    "id": "...",
    "username": "...",
    "email": "...",
    "roles": ["player"],
    "is_active": true
  }
}
```

## Configuration

The server reads configuration from `MythosEngine/config/settings.json` and `.env` files. Settings can be:
- Retrieved via `GET /settings`
- Updated via `PUT /settings`

Common settings:
- `VAULT_PATH` — Path to user vault
- `VAULT_TYPE` — Storage type (sqlite, hybrid, etc.)
- `COMPLETION_MODEL` — AI model for completion tasks (gpt-4o, etc.)
- `EMBEDDING_MODEL` — Embedding model (text-embedding-3-small, etc.)
- `MAX_TOKENS` — Max tokens for AI responses
- `THEME` — UI theme (Light, Dark)
- `FONT_SIZE` — UI font size

## AI Engine Integration

The AI engine is initialized on startup via:

```python
ctx.ai = ModelRouter(ctx.config, storage=ctx.storage)
```

If initialization fails, the server logs a warning and continues. AI endpoints will return `503 Service Unavailable` if the engine is not available.

The `ModelRouter` handles routing tasks (ask, summarize, suggest_tags, etc.) to configured backends based on `AI_BACKENDS` config.

## Performance Notes

- Tokens are stored in memory; they are lost on server restart
- For production, consider using Redis or a database for token storage
- The `list_*` methods in storage may be inefficient for large datasets; consider pagination
- AI requests may be slow depending on the configured backend
